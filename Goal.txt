## 📅 **Tydzień 2 – Zaawansowane API, autoryzacja i bazy danych**

### ✅ Dzień 5 – Uwierzytelnianie (NextAuth) i ochrona API

**Cel:** Dodać logowanie i zabezpieczyć wybrane endpointy.

- [ ] **Przeczytaj dokumentację NextAuth dla App Router:**  
  [NextAuth.js](https://next-auth.js.org/configuration/nextjs#in-app-router)
- [ ] **Zainstaluj NextAuth i skonfiguruj podstawowe logowanie** (np. przez GitHub/Google):
  ```bash
  npm install next-auth
  ```
- [ ] **Utwórz plik `app/api/auth/[...nextauth]/route.ts`** z konfiguracją (możesz skorzystać z przykładowego kodu z dokumentacji).
- [ ] **Dodaj provider logowania** (np. GitHub) – musisz utworzyć aplikację OAuth na GitHubie i dodać dane do `.env`.
- [ ] **Stwórz prosty komponent logowania** (przycisk "Zaloguj") – użyj `signIn` z `next-auth/react`.
- [ ] **Zabezpiecz stronę `/profile`** – sprawdź sesję po stronie serwera (`getServerSession`) i przekieruj niezalogowanych.
- [ ] **Zabezpiecz route handler `POST /api/posts`** – dodaj sprawdzenie sesji, aby tylko zalogowani mogli dodawać posty.
  ```ts
  const session = await getServerSession(authOptions);
  if (!session) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }
  ```
- [ ] **Przetestuj:** Spróbuj dodać post bez logowania – powinieneś dostać błąd 401.

### ✅ Dzień 6 – Autoryzacja i role użytkowników

**Cel:** Rozróżnić zwykłych użytkowników od adminów.

- [ ] **Rozszerz model Prisma** o pole `role` w modelu `User`:
  ```prisma
  model User {
    id        Int    @id @default(autoincrement())
    email     String @unique
    name      String?
    role      String @default("user") // "user" lub "admin"
    posts     Post[]
  }
  ```
- [ ] **Wykonaj migrację** (`npx prisma migrate dev`).
- [ ] **Stwórz endpoint `GET /api/users`** – dostępny tylko dla admina, zwracający listę użytkowników.
- [ ] **W route handlerze sprawdź rolę**:
  ```ts
  const session = await getServerSession(authOptions);
  if (!session || session.user.role !== 'admin') {
    return NextResponse.json({ error: 'Forbidden' }, { status: 403 });
  }
  ```
- [ ] **Dodaj możliwość promowania użytkownika na admina** (np. endpoint `POST /api/users/make-admin` – tylko dla istniejącego admina).
- [ ] **Przetestuj** różne scenariusze: logowanie jako user, jako admin.

### ✅ Dzień 7 – Relacje w bazie danych (Prisma)

**Cel:** Połączyć posty z użytkownikami (każdy post ma autora).

- [ ] **Zaktualizuj model `Post` w Prisma**:
  ```prisma
  model Post {
    id        Int    @id @default(autoincrement())
    title     String
    body      String
    userId    Int
    user      User   @relation(fields: [userId], references: [id])
    createdAt DateTime @default(now())
    updatedAt DateTime @updatedAt
  }
  ```
- [ ] **Wykonaj migrację**.
- [ ] **Zmodyfikuj `POST /api/posts`** – podczas tworzenia posta przypisz go do zalogowanego użytkownika (pobierz `userId` z sesji).
- [ ] **Zmodyfikuj `GET /api/posts`** – dołącz dane autora (`include: { user: true }`).
- [ ] **Dostosuj typy TypeScript** (możesz użyć wygenerowanych typów z Prismy).
- [ ] **Na stronie głównej wyświetl również autora posta** (np. `Autor: {post.user.email}`).

### ✅ Dzień 8 – Zaawansowane operacje na API (PUT, DELETE)

**Cel:** Pełny CRUD dla postów.

- [ ] **Dodaj route handler `PUT /api/posts/[id]`** – aktualizacja posta (tylko dla autora lub admina).
  - Sprawdź, czy post istnieje i czy użytkownik ma prawo go edytować.
  - Użyj `prisma.post.update`.
- [ ] **Dodaj route handler `DELETE /api/posts/[id]`** – usuwanie posta (tylko autor lub admin).
- [ ] **Dodaj przyciski edycji/usuwania na stronie `/profile`** (dla postów zalogowanego użytkownika).
- [ ] **Zaimplementuj formularz edycji** (może być na osobnej stronie lub w modalu).
- [ ] **Przetestuj** każdą operację.

### ✅ Dzień 9 – Walidacja danych (Zod)

**Cel:** Bezpieczna walidacja danych wejściowych w API.

- [ ] **Zainstaluj Zod**: `npm install zod`
- [ ] **Stwórz schematy walidacji** dla `POST` i `PUT` (np. `title` min 3 znaki, `body` min 10).
  ```ts
  import { z } from 'zod';
  const postSchema = z.object({
    title: z.string().min(3),
    body: z.string().min(10),
  });
  ```
- [ ] **W route handlerach użyj `postSchema.parse(body)`** – zwróci błąd, jeśli dane są niepoprawne.
- [ ] **Obsłuż błędy walidacji** – zwróć odpowiedź z kodem 400 i listą błędów.
- [ ] **Dodaj walidację dla query parameters** (np. przy filtrowaniu).

## 📅 **Tydzień 3 – Zaawansowana logika, filtrowanie, paginacja i webhooki**

### ✅ Dzień 10 – Filtrowanie, sortowanie i paginacja

**Cel:** Umożliwić pobieranie postów z parametrami.

- [ ] **W `GET /api/posts` odczytaj query parameters**:
  - `page` (numer strony)
  - `limit` (ilość na stronę)
  - `sortBy` (np. 'createdAt')
  - `order` ('asc'/'desc')
  - `authorId` (filtrowanie po autorze)
- [ ] **Zaimplementuj paginację** z użyciem `skip` i `take`.
- [ ] **Dodaj sortowanie** (przez `orderBy`).
- [ ] **Zwróć dodatkowe informacje** – całkowita liczba postów, liczba stron.
- [ ] **Przetestuj**: `/api/posts?page=2&limit=5&sortBy=createdAt&order=desc`

### ✅ Dzień 11 – Zaawansowane relacje i zliczanie

**Cel:** Dodać statystyki i agregacje.

- [ ] **Stwórz endpoint `GET /api/users/[id]/stats`** – zwraca:
  - liczbę postów użytkownika
  - datę rejestracji
  - ostatni post
- [ ] **Użyj Prisma do agregacji** (`_count`, `orderBy`, `take`).
- [ ] **Dodaj endpoint `GET /api/posts/popular`** – zwraca posty z największą liczbą komentarzy (jeśli dodasz model `Comment`).
- [ ] **Opcjonalnie:** Stwórz model `Comment` i powiąż z postami.

### ✅ Dzień 12 – Webhooki i integracje z zewnętrznymi API

**Cel:** Symulować webhook – np. powiadomienie o nowym poście.

- [ ] **Stwórz endpoint `POST /api/webhooks/new-post`** – będzie przyjmował dane z zewnętrznego serwisu (np. gdy inny system doda post).
- [ ] **Zabezpiecz go** prostym kluczem API w nagłówku (np. `X-Webhook-Secret`).
- [ ] **Odbierz dane, zapisz w bazie** (lub wyślij notyfikację).
- [ ] **Przetestuj** za pomocą `curl` lub Insomni.

### ✅ Dzień 13 – Testowanie API (Jest + Supertest)

**Cel:** Napisać podstawowe testy dla route handlerów.

- [ ] **Zainstaluj potrzebne pakiety**: `jest`, `supertest`, `@types/jest`, `ts-jest`.
- [ ] **Skonfiguruj Jest** dla Next.js (możesz skorzystać z oficjalnego przykładu).
- [ ] **Napisz testy dla `GET /api/posts`**:
  - sprawdź status 200
  - sprawdź, czy zwraca tablicę
- [ ] **Napisz testy dla `POST /api/posts`**:
  - z poprawnymi danymi – status 201
  - z błędnymi danymi – status 400
  - bez autoryzacji – status 401
- [ ] **Uruchom testy** i popraw ewentualne błędy.

### ✅ Dzień 14 – Podsumowanie i wdrożenie produkcyjne

**Cel:** Ostateczne szlify, refaktoryzacja i deployment.

- [ ] **Przejrzyj wszystkie route handlery** – czy obsługują błędy, czy mają odpowiednie kody HTTP.
- [ ] **Dodaj logowanie błędów** (np. do pliku lub usługi zewnętrznej).
- [ ] **Zadbaj o zmienne środowiskowe** – w `.env.production` ustaw prawdziwe dane bazy, klucze OAuth itp.
- [ ] **Wdróż na Vercel** (lub inny hosting) – pamiętaj o skonfigurowaniu zmiennych środowiskowych w panelu Vercel.
- [ ] **Przetestuj działanie w produkcji** – wszystkie endpointy, uwierzytelnianie, baza danych.
- [ ] **Odpowiedz sobie:** Czy jesteś w stanie samodzielnie zbudować pełną aplikację z backendem w Next.js? Czy rozumiesz każdy fragment kodu?
